# JCTM Digital Sanctuary

## Overview

This project is a full-stack pnpm monorepo for the Jesus Christ Temple Ministry (JCTM) in Warri, Nigeria. It aims to be a production-grade church platform offering a comprehensive digital experience, including sermons, AI-powered TempleBots, a giving portal, a real-time global altar counter, and various community engagement features. The vision is to provide a robust online presence that supports the ministry's global outreach and spiritual nourishment through technology.

## System Architecture

The project is structured as a pnpm monorepo with distinct packages for frontend, backend, database client, API types, and API hooks.

**Core Architectural Components:**

-   **Frontend (`@workspace/jctm-platform`):** Built with React 18, Vite 7, and Tailwind CSS v4, operating as a Single Page Application (SPA).
-   **Backend (`@workspace/api-server`):** Uses Express 5, Drizzle ORM, and Pino for logging.
-   **Database Client (`@workspace/db`):** Handles Drizzle schema definitions and connects to Neon PostgreSQL.
-   **API Types (`@workspace/api-zod`):** Shared Zod schemas for API validation.
-   **API Hooks (`@workspace/api-client-react`):** TanStack Query wrappers for data fetching.

**Event Promotion Engine (Time-Driven Lifecycle):**

-   **DB Table:** `event_promotions` — schema in `lib/db/src/schema/eventPromotions.ts`. Stores slug, title, subtitle, artwork, location, CTA, `start_at`/`end_at`, channel toggles (`show_banner`, `show_popup`, `show_sticky_bar`), and a `push_sent_at` lock that guarantees the LIVE-transition push fires exactly once.
-   **API Routes:** `artifacts/api-server/src/routes/eventPromotions.ts`
    -   `GET /api/event-promotions/active` — public, returns the highest-priority active promotion with computed `phase` (upcoming|live|ended), `msUntilStart`/`msUntilEnd`, plus `serverTime` for client clock-skew correction. Cache: 15s s-maxage + SWR.
    -   `GET|POST|PUT|DELETE /api/admin/event-promotions[/:id]` — protected by `requireAdminRole("livestream")`.
-   **Warri Crusade 2026 Admin Tile:** `WarriCrusadeStatsTile` (`artifacts/jctm-platform/src/components/admin/WarriCrusadeStatsTile.tsx`) is rendered at the top of the **Events** section in the Admin dashboard. It auto-refreshes every 30 s from `GET /api/admin/warri-crusade/stats` (livestream/sermon/gallery role required) and shows: campaign live/ended badge with countdown to end, four KPI cards (active subscribers, broadcasts sent, average delivery rate, click-through count + CTR%), the next scheduled half-hourly broadcast (relative + WAT clock time, snapped to the next `:00` or `:30` UTC slot), a 24-hour stacked-bar + line chart (sent vs failed bars, clicks line), and a list of the 20 most-recent `broadcast_events` rows for the campaign. Includes a **"Send broadcast now"** button (visible only while the campaign is active) that opens a confirmation modal showing the notification preview + active-subscriber count, then calls `POST /api/admin/warri-crusade/broadcast-now` (livestream role) which delegates to the exported `broadcastWarriCrusadeManual()` helper in `cron.ts`. The helper enforces a **5-minute cooldown** by querying `broadcast_events` for any prior row whose `message` carries the synthetic `[manual:…]` tag inside the cooldown window — duplicate clicks return HTTP 429 with the cooldown reason; successful sends fan out the same payload as the half-hourly cron and toast the result (delivered / failed / deactivated counts) on the admin dashboard. Half-hourly auto-broadcasts continue independently and use a different (`[slot:…]`) dedup key, so a manual send never blocks the next scheduled half-hour slot. Click-through tracking flows from the SW `notificationclick` handler — it fires a fire-and-forget POST to `/api/push/track-click` (with `keepalive`) before navigating, recording each click in the new `notification_clicks` table (`broadcast_type`, `target_url`, `visitor_id`, `clicked_at`) so CTR is computed as `clicks / total_sent` per campaign. The endpoint is unauthenticated by design (anyone tapping a notification triggers it) but values are length-clamped before insert. Index `notification_clicks_type_idx` keeps the per-campaign aggregations fast. Service worker bumped to `CACHE_VERSION = "jctm-v5"` so existing PWA installs pick up the new click handler on next activation.
-   **Warri Crusade 2026 Half-Hourly Multi-Channel Broadcast:** `checkAndBroadcastWarriCrusadeHalfHourly` in `artifacts/api-server/src/lib/cron.ts` is invoked from the existing per-minute scheduler tick. It fires **once per UTC half-hour slot** (a single fire when current minute is 0–4 OR 30–34) from now until the campaign end (`2026-05-01T21:00:00+01:00`), then disables itself permanently. Each fire dispatches (a) a web-push notification (rotating body picked deterministically from a 12-message pool keyed by `(hour * 2 + halfFlag)` so two consecutive sends never reuse the same body, `url: /crusade`, `data.broadcastType: "event_reminder"` — re-using the existing yellow "Coming Soon" SW relay palette so connected clients see an in-app sonner toast) via `dispatchPushNotification`, (b) an Expo push fan-out to all `expo_push_tokens.is_active = true` rows via `dispatchExpoPush`, and (c) inserts a `broadcast_events` row (`type: "warri_crusade_promo"`, `message` includes the half-hour slot key — `WARRI_CRUSADE_BODY [slot:YYYY-MM-DD-HH-MM]` where MM is `00` or `30`) which serves both as the **idempotency lock** (exact-match check before insert prevents duplicate sends within the same 30-min slot, even across process restarts) and as the source for the SSE missed-broadcast re-engagement card. Email is intentionally NOT included in this half-hourly fan-out to avoid spamming the devotion subscriber list — the existing daily devotion email pipeline remains the email channel of record. All dispatches are logged to `push_dispatch_log` via the standard pipeline.
-   **Production-grade Live Location Maps (April 2026):** Every venue map across the platform (Crusade page, Conference Registration page, Leadership/Contact page, and any future event page) now uses a single reusable `<VenueMap>` component (`artifacts/jctm-platform/src/components/VenueMap.tsx`) instead of static Google Maps `<iframe>` embeds. Engine: **Leaflet 1.9 + react-leaflet 5 + OpenStreetMap tiles** — industry-standard open-source map stack used by Wikipedia, GitHub, Foursquare, etc. No API key, no billing setup, no domain restrictions required, fully production-ready out of the box. **Architecture:** the heavy Leaflet payload (~150 kB) is split into its own chunk via `React.lazy` + dynamic `import("./VenueMapLeaflet")`, AND the chunk is only fetched once the map scrolls into the viewport via `IntersectionObserver` (200 px rootMargin pre-warm) — first-paint of the host page is completely uncoupled from map cost. **Features:** branded gold/sky/purple SVG pin (per-venue accent colour, no broken default-marker images), interactive popup with event name + date + address + Directions/Apple Maps buttons, footer row with Google Maps / Apple Maps / Waze deep-link buttons (every venue exports a `venueDeepLinks()` helper that builds all three URLs from one lat/lng + address pair), animated shimmer skeleton loader before tiles render, full error boundary that catches chunk-load or runtime failures and falls back to "address + Open in Google Maps" CTA so the user is never stuck with an empty box, mobile touch-friendly defaults (scroll-wheel zoom + dragging are disabled until the user explicitly taps the map, so the page never hijacks scroll), full keyboard navigation + ARIA `role="region"` + `aria-label` on the container, and a `venues` prop for multi-marker maps that auto-fits viewport bounds. **Venue registry** (`artifacts/jctm-platform/src/constants/venues.ts`) is the single source of truth for every event location — each `Venue` carries id, name, lat/lng (validated at runtime via `isValidVenue()`), address, optional `whenLabel` and `accent` colour, and optional `zoom` override. Three venues are pre-registered: `WARRI_CRUSADE_VENUE` (Ighogbadu Primary School, 5.5167°N / 5.7500°E, gold), `CHURCH_HQ_VENUE` (Ebrumede Roundabout, 5.5552°N / 5.7833°E, sky-blue), `CONFERENCE_VENUE` (Church Auditorium, 5.548°N / 5.773°E, purple). Adding a new event venue is now a 5-line constant edit. The Leaflet popup styling lives in `index.css` (`.leaflet-container`, `.leaflet-popup-content-wrapper`, `.venue-pin-wrapper`, `.leaflet-control-attribution`) so popups inherit the site's typography and dark palette without per-page overrides. Three packages added to the platform workspace: `leaflet@1.9.4`, `react-leaflet@5.0.0`, `@types/leaflet@1.9.21`.
-   **Generic Campaign Promotion Broadcast Mode (April 2026):** Any future event promotion can now run its own recurring multi-channel broadcast cadence directly from the admin panel — no code changes, no per-campaign hardcoded broadcaster needed. Six new columns on `event_promotions` (`broadcast_enabled`, `broadcast_cadence` ∈ `half_hourly|hourly|daily|custom`, `broadcast_interval_minutes` (5–1440, only for `custom`), `broadcast_messages` jsonb string array (≤24 entries, ≤240 chars each — rotates deterministically by slot), `broadcast_title_override`, `broadcast_image_url`) are added by an idempotent `ALTER TABLE … ADD COLUMN IF NOT EXISTS` migration in `artifacts/api-server/src/lib/migrations.ts`, gated by partial index `event_promotions_broadcast_enabled_idx ON (broadcast_enabled, end_at) WHERE broadcast_enabled = true`. **All existing rows (including Warri Crusade) default to `broadcast_enabled = false`**, so the legacy hardcoded `checkAndBroadcastWarriCrusadeHalfHourly` keeps running unchanged and there's no double-fire risk. The new `checkAndBroadcastCampaignPromotions` in `cron.ts` runs each per-minute tick, queries active enabled promotions whose `end_at > now`, and uses `computeBroadcastSlot(now, cadence, intervalMinutes)` to decide whether *this* minute is inside a fire window: half_hourly fires at minutes 0–4 and 30–34 UTC; hourly at minutes 0–4; daily at hour=0 minutes 0–4; custom fires `min(5, interval)` minutes into each `floor(epochMin / interval)` slot. Each fire dispatches (a) a web push via `dispatchPushNotification` (`url: ctaUrl`, `data.broadcastType: "event_reminder"` to re-use the yellow Coming-Soon SW relay), (b) an Expo push fan-out to all active tokens via `dispatchExpoPush`, and (c) inserts a `broadcast_events` row with `type: "campaign_promo"` and `message = "${body} [slot:${bucket}:promo${id}]"` — different namespace from `warri_crusade_promo`, so dedup never collides. Idempotency is the same SELECT-then-INSERT-as-lock pattern used by every other broadcaster, keyed off the synthetic slot tag. The notification body is picked by `pickCampaignBody(messages, slotIndex)`: rotates through `broadcast_messages` if any, otherwise falls back to `subtitle`, otherwise the title. Title and image fall back to `broadcastTitleOverride ?? title` and `broadcastImageUrl ?? artworkUrl`.
-   **Admin UI for Campaign Broadcasts:** The Event Promotions form in `Admin.tsx` (EventPromotionsSection) gains a collapsible **"Broadcast Cadence"** section below the channel toggles. Toggle to enable, then pick cadence from a dropdown (custom reveals an interval-minutes input with 5–1440 validation), enter optional title/image overrides, and one rotating-body per line in a textarea. The promotion row card shows a primary-tinted `Megaphone` chip with the cadence label whenever a campaign has broadcasts enabled, with a tooltip showing the rotating message count. The new fields flow through `validatePromotion` in `artifacts/api-server/src/routes/eventPromotions.ts` (server-side validation mirrors the client constraints), `serialize()` (fields surface on every promotion API response with safe defaults), and the existing PUT `Object.keys` loop (so partial admin patches just work).
-   **Backend Lifecycle:** `checkEventPromotionTransitions` in `artifacts/api-server/src/lib/cron.ts` runs every minute. Fires a "Now Live" web-push + logs to `broadcast_events` when an active promotion crosses its `start_at` (with `push_sent_at` as the idempotency lock). Also fires a one-shot 30-min "starting soon" reminder via `broadcast_events` de-dup.
-   **Recurring 6-hour Reminder Cron:** `checkAndSendEventReminders` (same per-minute tick) walks every active upcoming event and fires across all enabled channels at each crossed boundary in `[6, 12, 18, 24, 30, 36, 42, 48, 60, 72, 96, 120, 144, 168] h` before `start_at`. Catch-up window = 6 h (covers brief outages without resurrecting day-old reminders). Per-boundary idempotency = exact-match `(type='event_reminder', title, message)` row in `broadcast_events`, so a restart never duplicates. Stops at `start_at` (handed off to the LIVE-transition push). Capped at 168 h to avoid spamming for far-future events.
-   **Multi-channel fan-out:** Each reminder fires (a) web-push via `dispatchPushNotification` with `broadcastType: "event_reminder"`, (b) an in-app sonner toast (yellow "Coming Soon" variant) via the SW `BROADCAST_PUSH` relay in `BroadcastEngagementSystem`, (c) a 4-second yellow ring pulse on `EventBanner` triggered by a `jctm:event-reminder` `CustomEvent` the relay dispatches on `window`, plus (d) a fresh `broadcast_events` row that powers the missed-broadcast re-engagement card and SSE banner refresh.
-   **Frontend Hook:** `useActiveEventPromotion` polls `/api/event-promotions/active` every 30s and re-derives `livePhase` + countdown every second on the client clock so the LIVE flip is precise.
-   **Frontend Components** (`artifacts/jctm-platform/src/components/event-promo/`):

Uses localStorage with a 6-hour TTL for dismissal (`jctm:warri-crusade-2026:dismissedAt`) — never permanent. **Event Notification Scheduler (30-min, Multi-Channel, Idempotent):**

-   **DB Tables:** `event_notification_subscribers` (email opt-in list with `unsubscribe_token`, `is_active`, `source_page`) and `event_notification_dispatch_log` (per-event/milestone/channel idempotency log with `attempts`, `status`, `last_error`, success/failure counts) — schema in `lib/db/src/schema/eventNotifications.ts`. Unique index on `(event_id, milestone_hours, channel)` is the dedup key that survives restarts.
-   **Scheduler:** `artifacts/api-server/src/lib/event-notification-scheduler.ts` runs every 30 min (registered in `cron.ts` with a 30 s startup delay; `EVENT_NOTIFICATION_INTERVAL_MS = 30 * 60 * 1000`). Constants: `MILESTONES = [24, 12, 6, 1]` hours, `MAX_ATTEMPTS = 3`, `CATCHUP_HOURS = 1`. Each tick (a) scans `event_calendar` for events starting in the next 25 h, (b) for each `(event, milestone)` whose window has been crossed, claims-or-advances a log row per channel and dispatches, (c) re-runs failed rows with `attempts < 3`. Channels: web push (`push-manager`), Expo push (inline `ExpoPushMessage` like `cron.ts`), email (`sendEventNotificationEmail` in `email-engine.ts` with WAT date formatting + hero image + unsubscribe link, sent to all `is_active` subscribers), and SSE (`sseBroadcaster` event types `event_notification_dispatched` + `event_notification_tick` for the admin live view).
-   **Public API:** `POST /api/event-notifications/subscribe` (email + sourcePage opt-in) and `GET /api/event-notifications/unsubscribe?token=…` (one-click HTML response).
-   **Admin API:** `GET /api/admin/event-notifications/{log,upcoming,stats}`, `POST /run-now`, `POST /:id/retry` — all gated by `requireAdminRole("livestream")`.
-   **Admin UI:** `EventNotificationsSection` in `Admin.tsx` (Events tab, alongside `EventPromotionsSection`) shows stat tiles, an upcoming-events milestone grid, and the dispatch log table with per-row Retry buttons. Polls every 30 s and listens to `/api/sermons/stream` for live row highlights.
-   **Public Subscribe Widget:** `EventNotificationSubscribe` (`artifacts/jctm-platform/src/components/EventNotificationSubscribe.tsx`) is mounted on the Events page just below the upcoming events grid. Mirrors `DevotionEmailSubscribe` styling; posts to the public subscribe endpoint.
-   **Coexistence with existing reminders:** This system is the **email + admin-observable** channel for upcoming events. The pre-existing `checkAndSendEventReminders` in `cron.ts` continues to fire web-push + in-app sonner toasts at the broader `[6, 12, 18, 24, 30, 36, 42, 48, 60, 72, 96, 120, 144, 168] h` boundary set; the new scheduler adds email + Expo + structured admin observability at the tighter `[24, 12, 6, 1] h` milestones.

**Technical Implementations & Feature Specifications:**

-   **Database Schema:** Core tables include `sermon_data`, `testimonies`, `event_calendar`, `giving_logs`, `member_directory`, `member_auth`, `conversations`, `messages`, `crusade_registrations`, `knowledge_chunks`, `site_stats`, `daily_devotions`, `moment_comments`, `moment_engagements`, `moment_likes`, `broadcast_events`, `livestream_override_state`, and gallery/admin/push/blog/livechat support tables.
-   **Role-Based Admin System:** Three independent admin roles (`gallery`, `sermon`, `livestream`) secured with passphrase-based HMAC-signed JWTs, providing granular access control to protected routes.
-   **YouTube Sync Pipeline:** A three-layered system utilizing WebSub (instant push), RSS (5-min polling with immediate API enrichment), and YouTube Data API v3 (30-min full metadata refresh) for near real-time sermon synchronization and metadata management.
-   **Featured Sermon Pinning:** Admins can manually pin a sermon as the Latest Broadcast / Featured video via `POST /api/admin/sermons/:videoId/pin` (and `/unpin`). The `sermon_data.pinned_at` column drives priority ordering in `/api/sermons/featured`, and pinned videos suppress the daily YouTube auto-promotion for 30 days — solving cases where YouTube uploads multiple copies of the same broadcast with slightly different publish times.
-   **Gallery Feature:** Supports image uploads with client-side compression, server-side thumbnail generation, and GCS object storage. It includes dedicated API endpoints for managing gallery content and integrates with the homepage slideshow.
-   **Live Stream Stability:** Adaptive quality, network-aware playback, robust error handling with self-recovery for YouTube embeds, and real-time viewer count tracking via SSE.
-   **Monetized YouTube Embeds (`components/YouTubeEmbed.tsx`):** Single, canonical component used for every YouTube iframe across the platform. Always points at `youtube.com/embed/` (never `youtube-nocookie`, never proxied), and supports two modes: `facade` (lazy thumbnail + click-to-play, the default — gives YouTube a clean user-initiated playback so pre-roll ads serve normally) and `eager` (mounted immediately, used for live streams, sermon detail pages, and after explicit "Watch" actions). Standardized URL params (`rel=0`, `modestbranding=1`, `iv_load_policy=3`, `playsinline=1`, `enablejsapi=1`, `origin`, `widget_referrer`); muted-autoplay loops were removed because YouTube does not monetize muted views. Background promo videos on Home/Crusade/Conference now use the facade pattern, and the Moments/IntroVideos vertical short carousels use eager + 9/16 aspect ratio with autoplay tied to the active card. Each embed emits an IntersectionObserver impression and listens to YouTube IFrame API postMessage events (play/pause/q25/q50/q75/complete) which are batched via `lib/youtube-analytics.ts` (sendBeacon/keepalive fetch) to `POST /api/video-events`. The backend aggregates counts in the `video_event_counts` table and exposes `GET /api/video-events/top` for admin dashboards. Optional `emitSchema` prop renders a VideoObject JSON-LD block inline for rich-result eligibility.
-   **Broadcast Automation Engine:** Features an AI-powered rebroadcast curation system that selects the best sermon for rebroadcast based on algorithmic scoring (recency, view count, featured status, keyword relevance) and generates metadata using GPT-4o-mini.
-   **PWA Implementation:** Includes a service worker for offline capabilities, enhanced manifest with shortcuts, and push notification handling.
-   **SEO Architecture:** Comprehensive SEO with Schema.org JSON-LD (ReligiousOrganization, WebSite, BroadcastService, FAQPage, Event, Person, ImageGallery/CollectionPage), per-page canonical URLs, Open Graph, Twitter Cards, geo meta tags, and dynamic sitemaps for sermons and gallery.
-   **AI Features:**
    -   **JCTM Local AI Engine:** A custom, low-latency inference system for 18 specific JCTM intent types, using TF-IDF keyword scoring and confidence-gated routing.
    -   **OpenAI Integration (gpt-4o):** Used for daily devotions, prayer ministry, TempleBots (for complex queries), Sermon Assistant (RAG over JCTM sermons), Scripture Study, Spiritual Insight, Voice Chat, Translation, and Testimony Reflection.
    -   **Daily Devotion (plain text):** The Devotion page (`/devotion`) and the Home page's daily devotion section render as plain text only — no gradient cards, icons, image-share artwork, or animations. Visual-share generation (`html-to-image` / `toPng`) and per-day card themes have been removed. The fallback pool in `devotion-engine.ts` carries 40 hand-written devotions (40-day rotation) used whenever the AI generator (GPT-4o) is unavailable, ensuring users always receive a fresh daily entry.
-   **Unified Contact FAB (`VoiceTempleBots.tsx`):** A single collapsible floating action button in the bottom-right corner reveals a 5-icon stack: Chat with TempleBot (sky-blue, bottom-24), Voice Assistant (violet, bottom-40), WhatsApp (green, bottom-56), Phone (blue, bottom-72), and Zoom (Zoom-blue, bottom-[22rem]). Expand/collapse state is persisted in `localStorage` (`jctm:contactsExpanded`). A pulsing LIVE badge appears over the master FAB whenever a service is broadcasting (driven by `useLivestreamStatus()`). The "Chat with TempleBot" entry dispatches a `jctm:open-templebots` window event consumed by the standalone `<TempleBots />` widget in `Layout.tsx`, which has had its own floating launcher and scroll-aware search pill removed to avoid duplicate UI; smart notifications and predictive whispers continue to surface as standalone toasts that open the chat on tap.
-   **Security:** Implements Helmet for HTTP security headers, rate limiting, Gzip compression, scrypt password hashing, production-configured CORS, and 1MB JSON body limits.
-   **Performance:** Utilizes code splitting, lazy loading, TanStack Query for optimized data fetching, Vite asset optimization, and production-specific build configurations.
-   **Monetization:** Integrated AdSense with Google Consent Mode v2 for compliant, consent-gated ad rendering and specific placements across various pages.
